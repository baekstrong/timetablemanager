## A. 코치 시간표 — 진행 중/임박 수업 색 표시

- **위치**: `src/components/schedule/CoachSchedule.jsx`
- 오늘 요일 열에서, **수업 시작 30분 전 ~ 수업 종료 시점**까지 해당 교시 셀을 강조 표시.
  - 스타일: `--accent-10` 배경 + `1px solid var(--accent)` 보더 (디자인 시스템 준수, 그라데이션·그림자 금지).
- 기존 로직 재사용: CoachSchedule.jsx 270-278 부근의 `classStartMin/classEndMin` 계산(마지막 수업 강조용)과 동일한 방식. 단, 강조 구간은 `시작-30분 ~ 종료`로 정의.
- 교시별 수업 길이는 `mockData.js`의 PERIODS 시간 문자열 기준 (1·2·4·5·6교시 90분, 3교시 자율 120분).
- **60초 간격 타이머**(`setInterval`)로 현재 시각 state를 갱신해, 페이지를 켜둔 채로도 강조 셀이 자동으로 이동. 언마운트 시 클리어.
- 코치 모드에서만 적용. 학생 시간표는 변경 없음.

## B. 미결제 수강생 배지 (코치 시간표)

- **위치**: 코치 시간표 수강생 이름 칩 렌더링 지점 (`CoachSchedule.jsx` / `useScheduleCore.js`).
- 판정: 시트 K열(결제유무)이 `X`인 수강생. 파싱된 student 객체에서 `getStudentField(s, '결제유무')` 패턴으로 접근 (개행 변형 `결제\n유무` 대응).
- 표시: 칩 안에 `미결제` 상태배지 — 상태칩 패턴 `배경 var(--error)1A + 보더 1px solid var(--error)4D + 텍스트 var(--error)`.
- 빈 값은 미결제로 취급하지 않음 (`X`만 미결제).
- **학생 모드에는 절대 노출하지 않는다** (코치 분기 내에서만 렌더링).

## C. 수강생 검색창 (수강생 관리)

- **위치**: `src/components/StudentManager.jsx` 헤더(버튼 영역 근처).
- 검색 인풋 1개: 입력값으로 목록 실시간 필터.
  - 이름 부분 일치 (대소문자/공백 무관)
  - 전화번호(P열) 숫자 부분 일치 (하이픈 제거 후 비교)
- 빈 입력이면 전체 표시. 기존 이름순 정렬 유지. 결과 0건이면 "검색 결과 없음" 문구.
- 클라이언트 필터링만 — API 호출 없음.

## D. 수동 SMS 발송 + 발송 상태창

- **진입**: StudentManager 헤더에 "문자 보내기" 버튼 → 모달 (새 컴포넌트 `src/components/SmsSendModal.jsx`).
- **수신자 목록**: 현재 월 시트의 활성 수강생 (StudentManager가 이미 보유한 목록 재사용). 각 행에 이름 + P열 전화번호.
  - 전화번호 없음/형식 불량 → 체크박스 비활성 + `⚠ 번호 없음` 경고 표시.
  - 전체 선택 체크박스 (선택 가능 인원 기준, `28/30`처럼 카운트 표시).
- **메시지 입력**: textarea + 바이트 카운터(EUC-KR 기준 한글 2바이트). 90바이트 초과 시 "LMS로 발송됩니다" 안내 문구 (Solapi가 자동 전환).
- **발송**: 수신자별 성공/실패를 구분해야 하므로 기존 `sendSMS(to, text)`를 수신자별로 `Promise.allSettled` 병렬 호출 (`sendBatchSMS`는 그룹 단위 결과만 반환하므로 사용하지 않음). `smsService.js`에 `sendManualSMS(recipients, text)` 래퍼를 추가해 수신자별 결과 배열을 반환.
- **발송 상태창**: 발송 후 같은 모달이 결과 화면으로 전환.
  - 수신자별 ✓ 성공 / ✗ 실패(사유 텍스트) 목록.
  - 실패 1건 이상이면 "실패자만 재발송" 버튼.
- 발송 이력의 영구 저장은 하지 않음 (YAGNI — 필요 시 추후 Firestore 로그 추가).

## E. 비밀번호 변경 (수강생)

- **위치**: `src/components/StudentInfo.jsx` 하단(계약 이력 카드 아래)에 "비밀번호 변경" 카드.
- 입력: 현재 비밀번호 / 새 비밀번호 / 새 비밀번호 확인.
- **신규 함수**: `firebaseService.updateUserPassword(userName, currentPassword, newPassword)`
  1. `users/{userName}` 문서 조회 → `password !== currentPassword`면 에러 반환.
  2. 일치하면 `password` 필드 갱신.
- 성공 시 클라이언트에서 함께 갱신 (훈련일지 세션 공유 유지):
  - `localStorage.login_credentials`의 `password`
  - `localStorage.savedUser`의 `password`
- 검증: 새 비밀번호 최소 4자, 확인 불일치 시 에러 표시.
- **코치 빙의 모드(studentForce)에서는 카드 숨김.**
- 비밀번호 평문 저장 구조 자체는 이번 범위에서 바꾸지 않는다 (기존 구조 유지).

## F. 만석 슬롯 보강 대기 + 순차 SMS

### 신청
- 학생 시간표(`StudentSchedule.jsx`)에서 만석 슬롯 클릭 시 현재 `alert('만석입니다')`(258-259) → **"보강 대기 신청" 모달**로 교체.
- 검증은 기존 보강 신청과 동일: 주횟수 한도, 14일 이내, 홀딩 기간 불가, 본인 정규 수업 슬롯 불가, 원래 수업(이동할 수업) 선택 필수.
- 같은 학생이 같은 날짜+슬롯에 중복 대기 불가.

### 데이터
- **신규 Firestore 컬렉션 `makeupWaitlists`** (기존 영구 변경용 `waitlistRequests`와 분리):

```
{
  studentName, phone,            // phone은 신청 시점에 시트 P열에서 복사
  date, day, period, periodName, // 대기 대상 슬롯 (특정 날짜의 일회성 보강)
  originalClass: { date, day, period }, // 빠지는 원래 수업
  status: 'waiting' | 'notified' | 'accepted' | 'declined' | 'expired' | 'cancelled',
  createdAt, notifiedAt, respondedAt
}
```

### 자리 발생 감지 (트리거 지점)
1. 홀딩 처리 완료 시 (학생 HoldingManager / 코치 StudentManager 양쪽 경로)
2. 결석 처리 완료 시 (양쪽 경로)
3. 보강 취소 시
4. **백스톱**: 코치가 시간표를 열 때 1회 검사 (기존 waitlist 여석 자동 감지 useEffect 패턴 재사용 — 시트 직접 수정으로 자리가 난 경우 보완)

### 순차 알림 프로토콜
- 자리 발생 → 해당 날짜+슬롯의 `waiting` 대기열에서 **`createdAt` 선착순 1명에게만** SMS 발송 + `notified` 전환.
- SMS 문구에 반드시 포함: ① 날짜/교시 ② **"1시간 내에"** ③ **"앱 시간표에서 수락"**.
  - 예: `[근력학교] 6/13(금) 5교시에 보강 자리가 났습니다. 1시간 내에 앱 시간표에서 수락해주세요. 미응답 시 다음 대기자에게 넘어갑니다.`
- **수락**: 학생 시간표 해당 슬롯의 `보강승인중` 칩 클릭 → 수락/거절 모달(남은 시간 표시) → 수락 시 정식 보강(`makeupRequests`) 생성 + `accepted`. 수락 시점에 보강 검증 재실행(그 주 쿼터 초과 등이면 수락 불가 안내).
- **거절**: `declined` → 즉시 다음 `waiting` 대기자에게 SMS.
- **무응답 1시간**: `notifiedAt` + 1시간 경과 시 `expired` → 다음 대기자에게 SMS. 수업 시작까지 1시간 미만이면 남은 시간이 시한.
  - 만료 판정은 **lazy 방식**: 별도 서버 타이머 없이, 감지 시점(앱에서 대기열을 읽는 모든 지점 — 시간표 로드, 트리거 실행 시)에 `notifiedAt` 경과 여부를 확인해 `expired` 처리 후 다음 순번 발송.
- **공통 시한**: 어떤 상태든 해당 수업 시작 시각이 지나면 `expired`.
- 자리가 2개 이상 나면 여석 수만큼 순번대로 동시 `notified` 가능.

### 시간표 표시
- **학생 본인**: 대기 슬롯에 상태 칩 — `waiting`이면 `대기중`, `notified`면 `보강승인중`(클릭 → 수락/거절 모달).
- **코치**: 같은 슬롯에 `이름 (보강승인중)` 칩 — WeeklySchedule 수강생 상태에 `makeup-pending` 추가. `waiting` 상태도 코치에게는 `대기 N명` 정도로 노출.
- 수락 완료 시 기존 `makeup`(보강) 상태로 전환되어 표시.

## G. 업데이트 공지 자동화 (관리자봇)

### 스크립트 `scripts/post-update-notice.js` (Node, 로컬 실행)
- 인자: 제목, 본문 (CLI 인자 또는 임시 파일 경로).
- 동작:
  1. `posts`에서 `author == '관리자봇' && isUpdateNotice == true && 미삭제`인 글을 모두 **소프트 삭제** (게시판 기존 soft delete 필드 방식 그대로).
  2. 새 글 생성: `category: 'notice'`, `author: '관리자봇'`, `isUpdateNotice: true`.
- Firestore 접근: 앱과 동일한 웹 SDK 구성(`.env`의 VITE_FIREBASE_* 값)을 Node에서 로드해 사용 (Firebase Auth 미사용 구조라 클라이언트 SDK로 쓰기 가능).
- 게시판 UI 변경 없음 — 공지 카테고리에 그대로 노출되고, 기존 새 글 알림 점이 작동.

### 워크플로 규칙 (CLAUDE.md에 명문화)
- main 푸시(배포) 전, **수강생이 체감하는 변경**이면 Claude가 공지 초안(제목+본문)을 터미널에 제시하고 **백관장 승인을 받은 뒤에만** 스크립트 실행.
- 거절 시 공지 없이 배포만 진행.
- 내부 리팩토링·마이너 버그 픽스는 공지 제안 자체를 하지 않음.

- **에러 처리**: SMS 발송 실패는 사용자에게 사유와 함께 표시(D), F의 자동 발송 실패는 콘솔/Sentry 로깅 후 대기 상태 유지(다음 감지 시점에 재시도 가능).
- 구현 완료 후 CLAUDE.md의 해당 섹션(컬렉션 목록, 컴포넌트 목록, SMS 표 등) 갱신 필수.
