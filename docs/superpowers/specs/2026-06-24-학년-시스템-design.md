# 학년 시스템 (체력 성장 게이미피케이션) — 설계

작성일: 2026-06-24

## 한 줄 요약

훈련일지 누적 훈련량을 경험치(XP)로 환산해 **초·중·고·대 13단계 "학년"**으로 표시하는 과정 지향 성장 시스템. 기존 **티어**(이번 달 성실성)와 별개의 축이며, **PR 갱신은 학년과 무관한 축하 이벤트**로 둔다.

## 왜 만드나 / 두 축의 역할 분리

| 축 | 질문 | 성격 | 갱신 |
| --- | --- | --- | --- |
| **티어** (기존) | "요즘 얼마나 열심히 다니나" | 이번 달 스냅샷, **강등 있음** | 매달 재계산 |
| **학년** (신규) | "지금까지 얼마나 쌓았나" | 평생 누적, **우상향만** | 증분 누적 |
| **PR** (기존, 분리) | "내 기록이 깨졌나" | 순간 이벤트 | 측정 시 축하 |

게임으로 치면 티어 = 주간 랭크, 학년 = 계정 레벨, PR = 도전과제 달성 알림. 서로 안 싸운다.

> **YAGNI / 결정 사항**: 학년은 "실력(체중대비 1RM 보정)"이 아니라 **"과정(누적 훈련량)"**으로 매긴다. 실력은 PR 축하로만 다룬다. → 체중·성별 스트렝스 스탠다드 매핑 안 함(체중은 맨몸운동 환산용으로만 사용).

## 학년 체계

13단계 + 진척바.

```
초등 1 → 2 → 3 → 4 → 5 → 6 → 중등 1 → 2 → 3 → 고등 1 → 2 → 3 → 대학(졸업)
```

- 진척바 = 현재 학년에서 다음 학년까지 몇 % / 남은 XP(kg).
- 최고 단계 "대학"은 엔드게임(졸업) 목표로 작동.

## 경험치(XP) 정의

```
XP = Σ (보정 훈련량)   ← 전 훈련일지 기록 누적

기록 1건 훈련량 = 중량(kg) × 반복수
  · 맨몸운동: 중량 자리에 체중(kg) 대입
  · 여성: × 1.5  (성별 보정 계수, 고정 상수, 추후 숫자만 조정)
```

- 훈련량 합산 로직은 기존 `getAttendanceRanking`(firebaseService.js:1649)이 이미 `kg×reps`를 집계하므로 그 로직을 재사용·추출한다.
- **맨몸운동 판별**: 훈련일지 record 스키마에서 맨몸 여부/중량 0 표기를 확인해야 함 → 구현 시 record 구조 확인(미해결 항목). 체중은 `users/{이름}.bodyweight`에서 읽음.

### 학년 곡선 (XP → 학년) — 실데이터 보정 완료

게임처럼 **학년이 올라갈수록 다음 학년에 필요한 XP가 증가**(누진). 아래 임계값은 현재 수강생 6.5개월치 누적 XP 분포로 보정한 **초기 확정값**(단위 t = 1000kg, 각 학년 "진입" 누적 XP):

| 학년 | 진입 XP | 학년 | 진입 XP |
| --- | --- | --- | --- |
| 초등1 | 0 | 고등1 | 65t |
| 초등2 | 1t | 고등2 | 95t |
| 초등3 | 2.5t | 고등3 | 135t |
| 초등4 | 5t | 대학1 | 200t |
| 초등5 | 8t | 대학2 | 280t |
| 초등6 | 13t | 대학3 | 380t |
| 중등1 | 20t | 대학4(졸업) | 500t |
| 중등2 | 30t | | |
| 중등3 | 45t | | |

> 대학은 1~4학년 4단계(엔드게임). 현재 최고(심성희 194t)는 고등3이라 대학은 아직 도달자 없음 — 장기 목표.

**시뮬 결과(여성 ×1.5 적용, 55명):** 심성희 194t → 고등3(최고, 졸업 직전), 다음 그룹(현주은 등) 고등1, 중앙값대 중등1~2, 신규 초등1. 종 모양 분포, **졸업자 0명**(엔드게임 유지). 운영하며 숫자만 조정하는 손잡이.

## PR 축하 (학년과 분리)

PR 측정 제출(`submitPersonalBest`) 결과에 따라 축하 팝업만 띄운다. 학년 XP에는 영향 없음.

| PR 유형 | 조건 | 축하 |
| --- | --- | --- |
| 무게 갱신형 (`oneRM` 등) | 이전 best 중량 초과 | 🎉 "신기록!" |
| 반복 기준형 (`weightThenReps`) | 같은 중량서 이전 반복수 초과 | 🎉 "기록 갱신!" |
| 반복 기준형 | **통과 기준 도달** | 🏆 "다음 중량으로 올라가도 좋아요!" |

**통과 기준(종목별):**

| 종목 | 기준 |
| --- | --- |
| 12분 맨메이커 | 200회 |
| 5분 한팔 스윙 | 100회 |
| 5분 하이풀 | 100회 |
| 5분 스내치 | 100회 |

- 기준 맵은 종목명 키 상수(`src/data/` 또는 `grades.js` 인접). 
- ⚠️ **리스크**: 종목명 문자열 매칭이라 훈련일지 종목명이 바뀌면 깨짐. 1차는 상수로, 필요 시 코치 편집 UI는 후순위.
- `submitPersonalBest`는 이미 `{ updated }`(신기록 여부)를 반환 → 반환값을 확장(`{ updated, milestone }`)하거나 호출부(PRSubmitModal)에서 판정.

### 축하 팝업 위치/전달

- **제출 즉시 그 자리**: `PRSubmitModal` 제출 → 반환값 보고 **모달 위로 바로 축하 팝업**. 화면 이동 없음.
- **학생이 본인 입력**: 본인이 즉시 봄. 끝.
- **코치가 대신 입력**: 그 자리엔 학생이 없으므로, **학생 `users` 문서에 pending 축하 플래그**(예: `prCelebrationPending: { exercise, kind('record'/'milestone'), at }`)를 남겨 **학생 다음 접속 시 팝업** 후 해제. 티어 인트로 팝업(`tierIntroPending`)·월간 도장 팝업(`seenByStudent`)과 동일 패턴 재사용.

```

[━━━━━━━━━░░░] 다음 학년까지 1,240,000kg · 73%
```

- "근력학교 N년차" 표기는 **넣지 않음** (앱 출시 6개월이라 의미 없음).
- 인사말/경험치바 **탭 → 성장 화면(Ranking)으로 이동**.

### B. 게시판 이름 옆 뱃지

이름 옆에 **학년 뱃지(코발트)** + **티어 뱃지(초록 등 기존)** 둘 다 표시 (PostList/PostDetail/CommentItem). 너무 빽빽하면 추후 학년만 남기는 식으로 조정(1차는 둘 다).

- 학년 뱃지 컴포넌트 `GradeBadge.jsx`는 `TierBadge.jsx`를 거울삼아 작성.
- 이름→학년 맵은 `getGradeMap()`(티어의 `getTierMap()` 패턴, 5분 캐시)으로 Dashboard에서 읽어 prop 전달.

## 성능 — 읽기 폭증 방지 (핵심)

매 로드마다 훈련일지 전체를 합산하면 학생당 수백~수천 read → 폭발. **티어 시스템과 동일한 캐시·증분 패턴**으로 해결.

### `users/{이름}` 캐시 필드 (추가)

| 필드 | 의미 |
| --- | --- |
| `xp` | 누적 보정 훈련량(kg, 숫자) |
| `grade` | 현재 학년 key(파생값 저장, 게시판 뱃지용) |
| `gradeMonth` | 마지막 계산 'YYYY-MM'(백필 중복 방지, 티어의 tierMonth 패턴) |
| `gradeIntroPending` | 백필로 처음 부여된 학년 안내 팝업 대기 플래그 |
| `prCelebrationPending` | 코치 대신 입력한 PR 축하 대기 `{exercise, kind, at}` |
| `bodyweight` | (Phase 2) 맨몸운동 환산용 체중. Phase 1 XP는 kg×회만 사용 |

### 갱신 흐름 (계획 단계 단순화 — 증분 대신 "본인 기록 전량 재계산")

- **학생 접속**: `refreshStudentXP({userName, gender})` — **본인 records만** (`where userName == name`, 본인 훈련 이력 = 수백 건 이하, 최다 심성희 441건) 전량 읽어 XP 재계산 → `xp`/`grade` user 문서에 저장. 1회 로그인당 1회, 본인 데이터로 경계됨(폭증 아님). 항상 정확(수정/삭제 반영). `xpUpdatedThrough` 증분 불필요 → 제거.
- **코치 접속**: `backfillGradesForStudents(students)` — `backfillTiersForMonth` 패턴. 전체 records 1회 읽어 이름별 그룹핑 → batch write. 게시판 뱃지 전원 표시. (`gradeIntroPending=true` 표시 → 본인 다음 접속 시 인트로 처리)
- **표시(인사말·뱃지)**: user 문서의 `grade`만 읽음 → **추가 read 0**(로그인·티어가 이미 읽음). 게시판은 `getGradeMap()`(5분 캐시).

> **읽기 비용 정리**: 폭증의 원인은 "게시판 뱃지 위해 전원 기록 재합산"인데, 그건 user 문서의 캐시된 `grade`로 해결. 본인 인사말만 본인 기록을 재계산하며 그건 경계된 양. "평생 우상향" 컨셉 위배 가능성(전량 재계산이라 과거 삭제 시 내려갈 수 있음)은 실무상 무시(일지 과거 삭제 거의 없음). 정 필요하면 `max(저장된 grade, 계산된 grade)`로 단조 보장.

## 순수 로직 모듈 (테스트 대상)

`src/utils/grades.js` — `tiers.js`를 거울삼아 Firebase/DOM 비의존:

- `GRADES`: 13단계 배열 `{ key, label, group('elem'/'mid'/'high'/'univ'), index }`
- `xpToGrade(xp)`: 누적 XP → 학년
- `gradeProgress(xp)`: `{ grade, nextGrade, pct, remainingXp }`
- `recordVolume(record, { bodyweight, gender })`: record 1건 → 보정 훈련량 (성별 ×1.5, 맨몸 체중 환산)

테스트: `src/utils/grades.test.js` (vitest) — 경계 XP에서 학년 전환, 성별 보정, 맨몸 환산, 누진 곡선 단조성 검증.

## 미해결 / 구현 시 확정할 항목

1. ~~학년 곡선 상수~~ — ✅ 실데이터로 보정 완료(위 표). 운영하며 조정.
2. **훈련일지 record 스키마** — 맨몸운동 판별 방법 확인. (확인됨: record `{userName, date, sets:[{intensity:{value,unit}, reps:{value,unit}}]}`, 훈련량 = `intensity.unit==='kg' && reps.unit==='회'`인 세트의 `value×value` 합. 맨몸은 unit이 'kg'가 아니라 현재 집계서 제외 → 맨몸 환산은 unit 판별 추가 필요)
3. **체중 입력 UX** — 어디서 유도할지(내 정보 카드 or 훈련일지). 미입력 시 맨몸운동 XP 처리(기본값/건너뜀) 정책.
4. **학년 뱃지 디자인** — 코발트 단색 칩, 라벨 표기("고등2" 축약).
5. **PR 통과 기준 종목명 매칭** — 현재 종목명과 정확히 일치하는지 확인.
